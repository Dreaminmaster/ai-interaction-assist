# 技术架构

## 1. 总体目标
本系统是一个本地一键打开的 AI 学习工作台，要求具备：
- 本地桌面应用体验
- 主题化学习流程
- 长期记忆与学习档案
- 可替换的模型接入层
- 可扩展的资料挂载与复习机制

## 2. 推荐技术栈
### 前端
- React
- Vite
- Tailwind CSS
- Zustand 或 Redux Toolkit

### 桌面封装
- Tauri

### 后端
- Node.js
- Express 或轻量 API 服务

### 数据存储
- SQLite
- Drizzle ORM 或 Prisma

### 内容渲染
- Markdown 渲染
- Mermaid 流程图支持
- KaTeX 公式支持

## 3. 分层架构
建议拆为五层：

### 3.1 UI 层
负责页面呈现与交互。
主要页面包括：
- 首页
- 主题详情页
- 学习页
- 记录页
- 薄弱点页
- 资料页
- 复习页

### 3.2 应用流程层
负责控制学习流程与页面状态。
主要职责：
- 主题切换
- 会话切换
- 模式切换
- 当前学习步骤控制
- 本轮结束后的沉淀触发

### 3.3 学习引擎层
系统核心，负责：
- 生成学习地图
- 生成讲解内容
- 发起检测问题
- 诊断问题类型
- 生成纠正内容
- 生成阶段总结
- 更新主题档案

### 3.4 记忆与数据层
负责：
- 主题长期档案
- 学习记录
- 薄弱点记录
- 全局学习画像
- 资料挂载元数据

### 3.5 AI Provider 层
抽象模型调用接口，避免系统和单一模型绑定。
建议封装以下能力：
- generateLesson
- generateQuiz
- diagnoseAnswer
- summarizeSession
- updateTopicMemory

## 4. 模块划分建议
### 4.1 packages/ui
保存通用 UI 组件：
- 三栏布局
- 卡片组件
- 消息气泡
- 标签页
- 模式切换器
- 学习记录展示组件

### 4.2 packages/core
保存核心业务逻辑：
- 学习流程编排
- 问题类型判断
- 主题状态流转
- 模式规则定义

### 4.3 packages/ai-provider
封装模型调用层。
要求：
- 上层不直接感知具体模型来源
- 支持后续替换为 API、OAuth、中间代理或本地模型

### 4.4 packages/memory
封装记忆相关逻辑：
- 主题记忆更新
- 学习画像更新
- 薄弱点频率统计
- 下次入口生成

### 4.5 packages/db
封装数据库访问层与 schema。

## 5. 核心运行流程
### 新建主题
1. 创建主题档案
2. 生成初始学习地图
3. 生成首轮诊断问题
4. 进入学习页

### 单轮学习
1. 读取主题档案
2. 显示上次摘要
3. 设定本轮目标
4. AI 讲解一个点
5. AI 提问
6. 用户回答
7. AI 诊断问题类型
8. AI 定向纠正
9. 更新主题档案和学习记录

### 继续学习
1. 读取最近主题
2. 读取最近一轮学习记录
3. 展示上次学到哪里、哪里还薄弱、下次从哪里开始
4. 继续本轮学习

### 复习
1. 读取薄弱点和错题
2. 按优先级抽取不稳内容
3. 通过提问完成复习
4. 更新薄弱点状态

## 6. 模型接入策略
第一版建议设计为 Provider 抽象层，不直接和某一种授权方式绑定。

### 原因
- 降低耦合
- 方便更换模型
- 后续可接入多种后端
- 避免第一版被单一接口限制

## 8. 为什么推荐 Tauri
- 安装包小
- 启动轻
- 适合本地个人工具
- 便于读写本地文件
- 后续容易封装成桌面应用

## 9. 第一版开发重点
先不要追求页面炫技，优先做通这三项：
- 学习流程引擎
- 记忆引擎
- 主题档案引擎

这三项跑通后，系统就已经具备核心价值。
