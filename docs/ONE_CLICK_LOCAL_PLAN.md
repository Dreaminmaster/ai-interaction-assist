# 一键本地打开方案

## 1. 为什么要改路线
当前仓库已经有前端、provider、后端网关这些骨架，但如果要达到“像网页一样打开，又是一键本地使用”的体验，继续维持前端和后端分开启动会比较麻烦。

更适合的路线是：

**单体本地网页应用**

即：
- 只保留一个本地应用入口
- 启动一个本地 Node 服务
- 自动打开浏览器
- 页面和 API 由同一个本地项目提供
- 后续再打包成双击启动程序

## 2. 最简单的目标形态
用户双击一个启动器后，实际发生的是：
1. 本地服务启动
2. 自动打开 `http://127.0.0.1:8787`
3. 学习系统页面出现
4. 所有数据保存在本地
5. 页面内部可切 provider

这样对用户来说，它看起来就是“一个本地网页应用”，而不是要分别启动多个服务。

## 3. 推荐的简化架构
### 方案：单体 Node 本地网页应用
建议改成：

- 一个 `app/` 或 `local-app/` 目录
- Node/Express 提供 API
- 同时提供静态前端页面
- 一个启动命令同时完成服务启动和浏览器打开

结构示意：

```text
local-app/
├─ package.json
├─ .env.example
├─ src/
│  ├─ server.ts
│  ├─ routes/
│  ├─ providers/
│  ├─ analysis/
│  └─ storage/
└─ public/
   ├─ index.html
   ├─ app.js
   └─ styles.css
```

## 4. 为什么这条路更适合当前目标
### 优点
- 对用户来说更像“一键打开网页”
- 只需要一个本地进程
- 不需要再理解前端和后端分开跑
- 浏览器天然就是界面容器
- 后续也可以再打包成桌面程序

### 缺点
- 前端工程能力会比 React/Vite 方案简单一些
- 如果后期做很复杂的 UI，仍可能需要更强框架

但对第一版原型来说，这条路更符合“先用起来”。

## 5. 推荐的分阶段实施
### Phase A：先做单体本地网页 MVP
保留核心功能：
- 主题创建
- 主题切换
- 本轮学习
- 回答输入
- session 沉淀
- provider 切换
- 本地保存

### Phase B：做自动打开浏览器
启动本地服务后自动打开浏览器到固定地址。

### Phase C：再做真正的一键启动器
后续可以：
- Windows 用 `exe` 打包
- macOS 用 app bundle 打包
- 或者用 Electron/Tauri 套壳

## 6. 功能裁剪建议
为了尽快实现一键本地网页版，第一版建议裁剪为：
- 单页面
- 左侧主题区
- 中间学习区
- 右侧状态区
- 统一本地存储
- 一个 `/api/analyze-topic-answer` 接口

先不追求：
- 多版本 App 页面并存
- 复杂 hooks 分层
- 复杂 provider 工厂 UI
- 多入口原型页

## 7. 具体建议
### 现在最该做的不是继续扩很多原型页，而是开始收敛。
建议后续改成：

#### 一个主页面
- `index.html`
- `app.js`
- `styles.css`

#### 一个本地服务
- `server.ts`
- `routes/analyzeTopicAnswer.ts`
- `providers/openaiResponses.ts`
- `providers/oauthBackend.ts`
- `providers/mock.ts`

#### 一个启动命令
例如：
- `npm run local`

后续再把它做成双击启动。

## 8. 最终判断
如果你的核心要求是：
- 一键本地打开
- 看起来像网页
- 不想分开启动多个服务
- 先快速用起来

那么最合适的路线不是继续扩当前分离式结构，而是：

**收敛成一个单体本地网页应用，然后再做自动启动。**

这条路是可行的，而且更符合你现在的真实需求。
